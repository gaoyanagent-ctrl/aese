# AESE 2.0：从“业务场景模拟”升级为“企业生命周期仿真世界”

你的这次调整是正确的。现在的 AESE 如果只是把“客户需求—采购—生产—库存”绘制成二维流程，并驱动几张业务单据，它本质上仍是：

> **带动画的业务流程演示系统。**

而你真正想要的是：

> **从零创建一家企业，让企业在时间、空间、资源、资金、人员、设备、订单和外部环境约束下持续运行；IAOS 是这家企业使用的运营系统，而不是仿真的全部。**

这意味着 AESE 需要进行一次根本性的定位升级。

---

## 一、重新定义 AESE

### 正式定位

**AESE 是一个由人类与 AI Agent 共同参与的企业生命周期仿真环境。**

它模拟的不是某一条流程，而是一家企业从无到有、从筹建到运营、从增长到危机处理的完整过程，包括：

```text
创始设想
→ 投资与融资
→ 集团及法人设立
→ 选址与土地/厂房获取
→ 工厂建设项目
→ 办公及生产设施建设
→ 设备采购、安装、调试
→ 组织设计与人员招聘
→ 产品研发与工业化
→ 客户和供应商开发
→ 试生产与量产
→ 销售、计划、采购、生产、仓储、交付
→ 财务核算与资金管理
→ 质量、设备、EHS、人力资源管理
→ 扩产、搬迁、并购、经营危机
```

### AESE 与 IAOS 的关系

AESE 不应该只是 IAOS 的演示层，也不应该完全依附于 IAOS。

更合理的关系是：

| 系统                | 职责                   |
| ----------------- | -------------------- |
| **AESE**          | 模拟企业客观世界和运行结果        |
| **IAOS**          | 管理企业业务对象、流程、权限、任务和决策 |
| **Agent Runtime** | 让 AI Agent 扮演岗位并执行工作 |
| **2D 交互界面**       | 展示企业世界并允许人类参与        |
| **场景与剧情系统**       | 产生订单、异常、市场变化和经营挑战    |

可以理解为：

> AESE 负责“世界如何运行”，IAOS 负责“企业如何管理这个世界”。

# 二、最重要的架构变化：建立三个不同的状态世界

这是 AESE 从业务 Demo 走向真正企业仿真的关键。

## 1. 客观现实状态

表示企业真实发生了什么：

* 厂房是否建成；
* 设备是否安装；
* 操作工是否到岗；
* 仓库中实际有多少物料；
* 机器是否真的故障；
* 产品是否真的生产完成；
* 银行账户实际有多少钱；
* 客户是否真正收到了货物。

这是 **AESE World State**。

---

## 2. IAOS 管理记录状态

表示企业系统中记录了什么：

* 系统库存；
* 设备台账；
* 员工档案；
* 采购订单；
* 收货记录；
* 生产报工；
* 质量检验结果；
* 财务凭证；
* 应收应付余额。

这是 **IAOS Business State**。

---

## 3. 人员与 Agent 的认知状态

表示不同角色知道什么、相信什么：

* 计划员认为供应商会按期交货；
* 采购员已经知道供应商可能延期；
* 仓库实际收货了，但还没有录入系统；
* 设备已经出现异常振动，但设备主管尚未发现；
* 财务认为客户会在月底付款；
* 销售 Agent 知道客户可能临时增加需求。

这是 **Actor Knowledge State**。

---

## 三种状态之间允许不一致

真实企业中经常存在：

```text
实际库存 ≠ 系统库存
实际设备状态 ≠ 设备台账状态
真实交期风险 ≠ 计划员认知
人员实际能力 ≠ HR 档案中的技能等级
实际成本 ≠ 标准成本
实际产量 ≠ 已报工产量
```

这些差异不是系统错误，而是企业管理中最有价值的仿真内容。

AESE 不应让所有数据天然一致，而应让 IAOS 和 Agent 通过：

* 盘点；
* 检验；
* 数据采集；
* 审核；
* 对账；
* 追溯；
* 异常调查；
* Agent 分析；

逐步发现和纠正这些偏差。

这会使 AESE 真正具有企业运营深度。

## 2. 时间与事件仿真引擎

AESE 应采用以**离散事件驱动为主**的时间系统。

不同活动使用不同时间尺度：

| 活动    | 典型时间尺度  |
| ----- | ------- |
| 建设项目  | 天、周、月   |
| 招聘与培训 | 天、周     |
| 采购与物流 | 小时、天    |
| 生产作业  | 秒、分钟、小时 |
| 财务结算  | 日、月、季度  |
| 战略决策  | 月、年     |

系统需要支持：

* 暂停；
* 单步运行；
* 1 倍速、10 倍速、100 倍速；
* 运行到下一个关键事件；
* 运行到指定日期；
* 回放历史；
* 创建仿真分支。

例如：

> 在相同初始条件下，分别采用“加班生产”“外协生产”“推迟交付”三种决策，对比结果。

## 3. 企业创建与治理

企业不能一开始就天然存在。

需要模拟：

* 创建投资主体；
* 注入初始资本；
* 注册集团和法人公司；
* 设立董事会及管理层；
* 设计股权和治理结构；
* 开立银行账户；
* 建立预算；
* 设立组织部门；
* 分配管理职责；
* 建立授权和审批规则。

第一批 Agent 可以从这里开始参与：

* 创始人 Agent；
* CEO Agent；
* CFO Agent；
* 项目总监 Agent；
* 人力资源总监 Agent。

## 4. 工厂选址与建设项目

这是现有 AESE 中缺失的重要部分。

### 选址对象

每个候选地点具备：

* 土地或租赁成本；
* 距离客户和供应商的距离；
* 人力成本；
* 人才可获得性；
* 物流条件；
* 水、电、气和环保容量；
* 地方政策；
* 自然灾害风险；
* 建设周期。

### 工厂建设项目

需要建立完整项目管理模型：

```text
工厂建设项目
├── 立项与可行性研究
├── 总体预算
├── 场地设计
├── 厂房建设
├── 办公区建设
├── 水电气等公用工程
├── 生产线建设
├── 仓储物流建设
├── 质量实验室建设
├── 消防与EHS设施
├── 设备安装
├── 验收
└── 项目转固定资产
```

每项活动都消耗：

* 资金；
* 时间；
* 材料；
* 人员；
* 供应商能力；
* 管理资源。

延期和超预算将影响后续投产。

```text

→ 设备方案
→ 投资申请
→ 预算审批
→ 设备询价
→ 供应商选择
→ 采购合同
→ 制造与交付

→ 安装
→ 调试
→ 验收
→ 固定资产转固
→ 使用
→ 点检与保养
→ 故障维修
```

* 点检计划；

* 备件；
* 故障记录；

# 五、AI Agent 不再是“聊天角色”

AESE 中的 Agent 应是实际承担岗位责任的企业成员。

## Agent 基础模型

每个 Agent 至少需要以下结构：

```text
Agent Identity
Agent Role
Organization Position
Goals and KPIs
Authority
Budget Authority
Knowledge Scope
Skills
IAOS Capabilities
Inbox and Tasks
Working Calendar
Memory
Decision Policy
Current Workload
```

## Agent 的完整工作循环

```text
观察企业状态
→ 获取IAOS信息
→ 接收任务和消息
→ 判断问题
→ 制订行动方案
→ 请求审批或直接执行
→ 调用IAOS Capability
→ AESE计算实际结果
→ Agent观察结果
→ 更新认知和后续计划
```

例如采购 Agent 不只是回答“是否应该采购”，而是：

1. 读取 MRP 和库存；
2. 检查预算；
3. 比较供应商价格、质量和交期；
4. 创建询价；
5. 组织比价；
6. 发起审批；
7. 创建采购订单；
8. 跟踪交货；
9. 处理供应商延期；
10. 根据履约情况更新供应商评价。

---

## 人类和 Agent 使用同一套岗位与能力体系

人类用户可以：

* 扮演董事长或投资人；
* 接管某个具体岗位；
* 与 Agent 共同组成管理团队；
* 审批 Agent 的决策；
* 向 Agent 分配任务；
* 替换某个 Agent；
* 观察 Agent 的决策过程；
* 修改组织结构和授权规则。

不应为人类和 Agent 建立两套系统。

两者都应通过 IAOS 的：

* 权限；
* Capability；
* 工作流；
* 任务；
* 消息；
* 审计；

执行工作。

# 六、业务场景仍然保留，但成为企业世界的一部分

现有“客户需求到采购、生产、库存”的场景不需要废弃。

它应从 AESE 的主体降级为三级内容体系中的一部分。

## 第一层：企业生命周期 Campaign

例如：

> 华辰热管理集团从创业到首个工厂量产。

跨度可能是两到三年。

## 第二层：经营阶段 Scenario

例如：

* 建设苏州工厂；
* 获取第一个新能源汽车客户项目；
* 建立供应链；
* 完成新品 APQP；
* 实现量产爬坡；
* 处理现金流危机；
* 建设第二工厂。

## 第三层：业务事件 Story/Event

例如：

* 客户需求增加 20%；
* 关键设备延期到货；
* 供应商出现批量质量问题；
* 试生产直通率低于目标；
* 核心工程师离职；
* 客户要求提前 SOP；
* 银行缩减授信；
* 设备突然故障。

这样，当前的业务场景模拟会成为一个更大企业世界中的局部剧情。

# 七、仿真必须遵守资源和经济约束

AESE 不能依赖随意修改数字来制造结果。

需要建立一套“企业运行守恒规则”。

## 资金约束

* 支出必须有资金来源；
* 设备采购会减少现金或增加负债；
* 工资、租金、材料和能源持续消耗现金；
* 收入只有在交付、开票和回款后逐步形成；
* 利润不等于现金；
* 扩张可能带来现金流危机。

## 物料约束

* 物料不会凭空产生；
* 投入、产出、废品和损耗必须平衡；
* 实际库存和系统库存可以不一致；
* 批次、物流单元和在制品必须有空间位置。

## 产能约束

生产能力由以下因素共同决定：

```text
设备能力
× 人员能力
× 班次时间
× 工装模具
× 材料可用性
× 工艺条件
× 质量水平
```

不是创建生产订单后就自动完工。

## 人员约束

* 人员有工作时间和负荷；
* 缺乏技能不能执行某些任务；
* 培训需要时间和成本；
* 关键岗位缺失会造成组织瓶颈；
* 人员会请假、离职、犯错或成长。

# 八、AESE 如何反向驱动 IAOS

你的新策略不应再是：

> 先完善 IAOS，再寻找业务场景验证。

而应改为：

> 建立华辰热管理集团，每当企业运行需要一种管理能力时，再为 IAOS 增加对应能力。

## 需求产生方式

例如，华辰热管理准备建设苏州工厂：

| 企业真实需求 | IAOS 被迫补充的能力   |
| ------ | -------------- |
| 工厂建设立项 | 项目管理、投资申请      |
| 预算控制   | 预算、费用、合同、付款计划  |
| 设备采购   | 采购、固定资产、设备项目   |
| 招聘核心团队 | 组织、岗位、编制、招聘    |
| 建立生产线  | 工厂资源、工作中心、设备   |
| 开发产品   | 产品、BOM、工艺、APQP |
| 试生产    | 生产、质量、追溯、异常    |
| 正式交付   | 销售、计划、WMS、物流   |
| 月末结算   | 应收、应付、成本、总账接口  |

这样 IAOS 的每一个功能都有明确的企业来源，而不是从 ERP、MES 产品功能清单中机械复制。

---

## 建议建立“能力缺口台账”

每次 AESE 中的角色无法完成工作时，记录：

```text
企业活动
当前目标
执行角色
所需业务对象
所需 Capability
当前 IAOS 是否支持
暂时人工处理方式
IAOS 后续实现优先级
```

这张台账将成为 IAOS 最真实的产品 Backlog。

# 九、推荐的第一个完整纵向场景

不要一开始模拟整个华辰集团。

第一个版本建议实现：

# **从零创建苏州工厂，到完成第一批产品交付**

可以命名为：

> **Project Genesis：华辰苏州工厂诞生计划**

完整过程如下。

## 第一阶段：成立公司

* 投资人注资；
* 注册集团及苏州制造公司；
* 任命 CEO、CFO 和工厂项目负责人；
* 建立初始组织；
* 制订三年经营计划；
* 建立初始预算。

## 第二阶段：建设工厂

* 选择厂址；
* 签订土地或租赁协议；
* 创建工厂建设项目；
* 设计厂区和车间布局；
* 建设厂房和公共设施；
* 采购办公及生产设施。

## 第三阶段：建设生产能力

* 确定首个产品：电池冷却板；
* 设计工艺路线；
* 计算设备和产能需求；
* 采购、安装和调试设备；
* 建立原材料仓、线边库和成品仓；
* 建立质量实验室。

## 第四阶段：组建团队

* 设计组织和岗位；
* 招聘厂长、计划、采购、质量、工艺、设备和仓储人员；
* 招聘操作工和检验员；
* 完成培训和岗位认证；
* 建立班次。

## 第五阶段：获得客户项目

* 接收客户 RFQ；
* 进行产品和工艺评估；
* 建立成本和报价；
* 获得定点；
* 创建客户项目；
* 执行 APQP 和试制。

## 第六阶段：试生产和量产

* 建立供应商；
* 采购首批物料；
* 到货和来料检验；
* 试生产；
* 处理设备、工艺和质量问题；
* 达到目标节拍和良率；
* 客户 PPAP 批准；
* 进入 SOP。

## 第七阶段：第一次完整交付

* 接收正式订单；
* 运行计划和 MRP；
* 采购和生产；
* 入库和发货；
* 客户收货；
* 开票；
* 回款；
* 计算首批产品实际成本和项目盈亏。

这一个纵向场景就足以持续驱动 IAOS 很长时间。

# 十、第一阶段不应追求的内容

目前暂时不要投入过多精力在：

* 复杂 3D 模型；
* 高精度物理仿真；
* 每台设备完整数字孪生；
* 所有 ERP、MES、PLM、财务模块；
* 数百个 Agent 同时自治；
* 过度复杂的宏观经济模型；
* 一开始模拟集团所有工厂。

第一阶段重点应是：

```text
可创建企业
可推进时间
可建设工厂
可消耗资金和资源
可招募人员
可安装设备
可形成生产能力
可获得客户订单
可完成第一批交付
可由人类和Agent共同运营
可通过IAOS完成管理
```

# 十三、核心设计原则

后续 AESE 设计应始终遵守以下原则：

1. **企业先于系统。**
   先确定企业需要做什么，再决定 IAOS 提供什么。

2. **现实状态不等于系统记录。**
   保留数据差异，才能模拟真实管理问题。

3. **Agent 必须承担岗位责任。**
   不能只做问答助手。

4. **所有行为都消耗资源。**
   时间、资金、人员和产能不能无限。

5. **所有决策都产生后果。**
   后果由仿真引擎计算，不由剧情直接指定。

6. **业务场景嵌入企业生命周期。**
   不再把单条流程当成整个 AESE。

7. **采用不同仿真粒度。**
   建设按天，运营按小时，设备过程可按分钟或事件计算。

8. **人类和 Agent 使用同一套 IAOS 能力。**
   统一权限、流程和审计。

## 最终建议

AESE 下一阶段不应继续完善现有“订单到生产”的二维业务图，而应先建立四个基础能力：

```text
1. 企业世界模型
2. 时间与事件仿真内核
3. 企业创建及工厂建设系统
4. 人类与Agent统一角色运行机制
```

然后以：

> **华辰热管理集团苏州工厂从成立到第一次客户交付**

作为第一个完整纵向版本。

从这个版本开始，IAOS 不再由抽象架构和功能清单推动，而由一家正在被真正创建和运营的虚拟企业持续提出需求。这样形成的 IAOS，会更接近你最初设想的“工业企业操作系统”，而不是另一个低代码 ERP/MES 平台。
