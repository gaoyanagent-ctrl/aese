# AESE 拥有只读 2D 场景预览界面

## 背景

ADR-001 将业务 UI 和运行期查询能力归属 IAOS。这一边界适合正式运行环境，但如果必须等异常事件、Agent 和 IAOS 查询合同全部完成后才开始 2D 沙盘，用户将长期看不到 AESE 的产品形态，也无法尽早验证信息布局、事件叙事和交互方式。

## 决策

AESE 仓库可以拥有一个只读的 2D 场景预览器，职责限定为：

- 读取 AESE 版本化场景数据并呈现企业、工厂、产线、对象、事件和预期 Agent 建议。
- 确定性播放、暂停、单步和重置演示故事。
- 为 IAOS 后续 API/SSE 接入定义前端数据源接口和视图合同。
- 用于产品验证、演示排练和场景包回归测试。

该界面不得：

- 自建业务数据库、工作流、权限、审计或 Agent 运行时。
- 修改正式业务对象，或把前端状态当作 IAOS 运行结果。
- 绕过 IAOS 受治理接口直接写数据库或发布正式事件。
- 演变为第二套 ERP/MES 操作台。

正式在线模式仍由 IAOS 提供业务运行时和查询接口。预览器通过 `ScenarioDataSource` 适配层切换静态场景数据与 IAOS API/SSE，组件不得直接依赖任一后端 DTO。

## 结果

正面影响：

- 可以在 3 到 4 个工作日内得到第一版可见、可操作的 2D 企业沙盘。
- 可以在 M4/M5 前验证七幕故事、22 个事件和关键 KPI 的表达是否有效。
- 静态预览与在线运行共用视图模型，后续接真数据时不重写页面结构。

代价和约束：

- 首版展示的是确定性故事回放，不等同于实时离散事件仿真。
- 必须持续标识 `Preview` 与 `Live` 数据源，避免误认静态结果为实时计算。
- 场景数据和 UI 视图模型之间需要维护转换与回归测试。

## 被否决方案

1. 等 M4、M5 完成后再做 M6：首次可见成果过晚，无法尽早验证产品形态。
2. 在 IAOS frontend 直接硬编码华辰故事：平台与行业样例耦合，不利于场景独立版本化。
3. 为预览器新建独立业务后端：重复 IAOS 运行时，违反 ADR-001。
